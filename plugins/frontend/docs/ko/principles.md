# Principles

이 문서는 "코드를 어디에 두는가"라는 질문에 답한다. 레이어 구조는 어휘이고, 이 문서의 원칙이 그 어휘를 실제 판단에 연결한다.

---

## Page-First

### 왜

처음부터 코드를 여러 레이어에 분산하면 두 문제가 동시에 생긴다. 하나는 **예측 추출** — 아직 재사용이 존재하지 않는데 "나중에 쓸 것 같아서" 공통 레이어로 올리는 것이다. 이렇게 올라간 코드는 실제 사용처가 한 곳이면서도 공통의 제약을 받게 되어, 가까이 있었다면 자연스럽게 진화했을 인터페이스가 일찍 굳는다. 다른 하나는 **책임 파편화** — 한 화면을 이해하려면 여러 폴더를 오가야 하는 상태다. 화면과 그에 필요한 로직이 붙어 있지 않으면, 코드를 읽는 사람도 수정하는 사람도 매번 퍼즐을 맞춰야 한다.

Page-first는 이 두 문제를 한 번에 푼다.

### 무엇

**Page는 단순한 라우트 엔트리가 아니라 화면 단위의 자족적 모듈이다.** 전용 상태·훅·컴포넌트·도메인 로직을 page가 **직접 소유**한다. 공통 레이어에 선제적으로 두지 않는다.

모든 코드는 page 안에서 시작한다. business-agnostic한 코드도 마찬가지다 — Button이나 useDebounce 같은 범용 코드라도 한 page에서만 쓰인다면 그 page 안에 둔다.

예외는 **page가 동작하기 위해 page보다 먼저 존재해야 하는 기반**뿐이다. 환경변수 정의, 라우트 경로 상수, 번역 설정, API 클라이언트, 정적 리소스, 디자인 시스템/헤드리스 컴포넌트, TanStack Query 팩토리가 여기에 해당한다. 이들은 page가 호출하는 대상이기 때문에 처음부터 `shared`에 둔다.

### 어떻게

page 내부에서 복잡도가 늘면 다음 흐름을 따른다.

```txt
page 하나로 시작
→ 필요하면 큰 단위 컴포넌트로 분해
→ 그 큰 단위도 필요하면 내부에서 다시 분해
→ 가장 작은 부분에서 재사용이 보이면 추출 검토
→ 조건을 모두 만족하면 다른 레이어로 이동
```

분해는 파일 길이가 아니라 **독립적 책임의 수**를 기준으로 판단한다. 한 파일 안에서 서로 무관한 상태나 흐름이 공존하거나, JSX와 로직 책임이 여러 개 섞여 있을 때가 분해 시점이다. 깊이 자체는 문제가 아니다. page 내부에서 계속 분해해도 된다.

## 추출과 이동

코드를 옮기는 행위는 두 단계로 나뉜다. **추출**과 **이동**은 다른 의미다.

### 추출

**추출은 Slice 내부에서 공통 코드를 분리하는 것이다.** Slice 내부의 여러 하위 모듈이 같은 코드를 쓰게 되면, 가장 가까운 공통 범위로 분리한다. 추출된 파일은 여전히 같은 Slice에 속한다.

```txt
[slice]/
├─ component-a/    ← 둘 다 filter-chip을 쓴다
├─ component-b/    ←
├─ _ui/
│  └─ filter-chip.tsx   ← Slice 내부의 가까운 공통 범위로 추출
└─ index.tsx
```

추출된 파일이 여러 개 생기면 `_ui/`, `_context/`, `_lib/` 같은 private 폴더로 묶는다. 처음부터 만들지 않고, 추출이 실제로 발생한 뒤에 묶는다.

### 이동

**이동은 다른 레이어로 옮기는 것이다.** Slice 간 재사용이 실제로 생기고, Slice 문맥 없이도 독립적으로 동작할 때 검토한다.

이동은 신중해야 한다. **추출이 기본이고, 이동은 최종 단계다.** 공통이 생겼다고 해서 곧바로 `shared`나 `widgets`·`features`·`entities`로 올리지 않는다. 먼저 가장 가까운 공통 범위로 추출하고, 그래도 해결되지 않을 때만 이동을 고려한다.

---

## 이동 조건

이동하려면 다음 세 가지가 **모두** 만족되어야 한다. 하나라도 불충족이면 현재 범위에 남긴다.

### 1. 2개 이상의 Slice에서 실제로 사용하고 있다

예측이 아닌 실제 사용이다. "두 번째 사용처가 생기면 쓸 것 같다"는 아직 이동 조건이 아니다. 두 번째 사용이 실제로 발생한 시점에 이동을 검토한다.

"2개 이상"의 기준은 대상 레이어의 허용 상위에 따라 달라진다 — 계급 전체는 [foundations](foundations.md#계급과-방향) 참조. `widgets`는 2개 이상의 page에서, `features`는 2개 이상의 page나 widget에서, `entities`는 2개 이상의 page·widget·feature 중에서, `shared`는 어느 상위 레이어의 Slice든 2개 이상에서 사용되어야 한다.

상위 선택 레이어가 아직 도입되지 않았다면 그 항목은 조건에서 제외된다. 예를 들어 `widgets`가 없는 프로젝트에서 `features` 도입 조건은 "2개 이상 page에서 반복"으로 줄어든다.

### 2. 책임이 안정적이다

인터페이스(props, 파라미터, 반환 타입)가 최근 변경되었거나 변경 예정이라면 이동하지 않는다. 이동한 뒤에 인터페이스가 흔들리면 여러 Slice의 코드를 따라 고쳐야 하고, 이 비용이 이동 이득을 상쇄한다. 필요하면 `git log`로 최근 변경을 확인한다.

### 3. Slice 문맥 없이도 의미가 성립한다

입력/출력이 특정 Slice 상태나 context에 결합되어 있으면 이동하지 않는다. 도메인 엔티티 타입을 직접 props로 받거나, 특정 Slice의 상태를 내부에서 참조하는 코드는 Slice 문맥에 묶여 있다. 범용 props와 인터페이스만으로 동작해야 이동 대상이 된다.

차이는 props 시그니처에서 드러난다.

```ts
// ❌ Slice 문맥에 결합 — ProductDetailState를 직접 요구
type Props = {
  state: ProductDetailState;
  onToggleSection: (section: SectionId) => void;
};
```

```ts
// ✓ 범용 인터페이스 — 어느 Slice에서든 사용 가능
type Props = {
  expanded: boolean;
  onToggle: () => void;
};
```

위 예시처럼 원시 타입과 범용 콜백만 받아 동작하는 형태가 되어야 다른 Slice에서도 의미를 유지한 채 재사용된다.

## 판단 순서

코드를 어디에 둘지 결정할 때 항상 다음 순서를 따른다.

1. **이 코드는 특정 page 전용인가?** → page에 둔다.
2. **더 작은 범위에서 해결할 수 있는가?** → 가장 좁은 범위(page 내부 컴포넌트, Slice 하위 폴더 등)에 둔다.
3. **공통이 생겼는가?** → 가장 가까운 공통 범위로 추출한다.
4. **2개 이상의 상위 Slice에서 실제로 사용하고 있는가?** → 아니면 현재 범위에 남긴다.
5. **Slice 문맥 없이도 의미가 성립하는가?** → 적절한 레이어로 이동한다.
6. **위 조건에 해당하지 않으면** 로컬에 남긴다.

이 순서는 "어떻게 하면 확장 가능한가"가 아니라 **"어떻게 하면 섣불리 움직이지 않는가"**를 지향한다. 로컬에 남기는 것이 기본이고, 이동은 조건을 모두 만족했을 때의 예외다.

---

## 이동 대상 분류

이동 조건을 모두 만족했을 때, 책임의 성격에 따라 목적지가 달라진다.

| 책임의 성격 | 이동 대상 |
|-------------|-----------|
| 복합 UI 블록 (여러 하위 조합으로 구성된 UI) | `widgets` |
| 사용자 인터랙션 단위 (UI + 로직) | `features` |
| 도메인 UI / 표시 로직 (label·color 매핑, 검증) | `entities` |
| business-agnostic 유틸·훅·UI | `shared` |

같은 코드를 이동할 때도 무엇을 하는 코드인가에 따라 다른 레이어로 간다. 도메인 용어가 포함된 UI는 `shared/ui`에 두지 않는다 — `shared/ui`는 business-agnostic 컴포넌트만 담는다.

---

## 이동이 항상 답은 아니다

같은 레이어의 Slice 간 조합이 필요한 상황이 생기면, 코드를 이동하는 대신 **사용하는 쪽에서 조합**하는 것이 답일 때가 많다.

header widget과 sidebar widget이 연동되어야 할 때, sidebar를 다른 곳으로 옮기는 것이 아니라, 두 widget을 사용하는 page가 상태를 소유하고 props로 내려준다. widget 자체를 이동하거나 widget 간 의존을 만들지 않는다.

같은 원리가 feature와 entity에도 적용된다. feature 간 의존이 필요하면 feature를 사용하는 쪽(page, widget)이 조합한다. entity 간 조합이 필요하면 entity를 사용하는 쪽(page, feature, widget)이 조합한다. **조합 책임은 상위 레이어에 있다.**

---

## 정리

- Page는 화면 단위의 자족적 모듈이다. 전용 상태·훅·컴포넌트·로직을 직접 소유한다.
- **가장 좁은 범위에서 시작**한다. 공통이 생기면 가장 가까운 공통 범위로 먼저 추출한다.
- **추출은 Slice 내부**, **이동은 다른 레이어**. 이동은 최종 단계다.
- 이동 조건: (1) 2개 이상 Slice에서 실제 사용, (2) 책임 안정성, (3) Slice 문맥 독립성. 셋을 모두 만족해야 이동한다.
- 같은 레이어 Slice 간 조합이 필요하면, 이동 대신 상위 레이어가 조합한다.
